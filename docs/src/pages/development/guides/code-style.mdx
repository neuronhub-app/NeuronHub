## Primary Directives

Those are engineering constraints you SHALL apply on EVERY line you write.

- "Simplicity is a prerequisite for reliability." - Dijkstra
- "There are two ways of constructing a software design: One way is to make it so simple that there are obviously no deficiencies, and the other way is to make it so complicated that there are no obvious deficiencies." - Hoare
- "Debugging is twice as hard as writing the code in the first place. Therefore, if you write the code as cleverly as possible, you are, by definition, not smart enough to debug it." - Kernighan
- "A complex system that works is invariably found to have evolved from a simple system that worked." - Gall

## Secondary Directives

The rules below are heuristics, not laws. They serve the Primary Directives. Any heuristic that increases complexity instead of reducing it has been misapplied.

### Manage complexity

You're working on a pre-MVP pre-PMF codebase:
- You must minimize surface area - every line of code is tech debt.
- Extreme complexity engineering is a must for survival.

### Write simple, self-documented code

Every dev overestimates their intelligence due to empathy gaps - KISS.

Comments:
- Every comment is a tech debt - it'll fuck you up once outdated.
- Exceptions are possible, eg:
    - Use only for unusual cases - tell why, not what.
    - Reasons for future removal (ie it covers a bug or tech debt).
    - [#Tags]. You SHALL preserve the `#AI` and `#AI-slop` tags.
    - etc.

### Best practices

In the order of importance:
- You must write using the inverted pyramid - declarations below their usage, not prior, etc.
- You must pursue __scope minimization__:
    - Keep live range low - declare at the usage.
    - Keep single-use declarations in their scopes/functions/file.
    - Inline the single-use vars as named args/kwargs.
- 2+ uses of a magic string → define as a var.
- Functions > 20 lines → decompose. Think of later debugging prod at 3am.
- Define explanatory variables (or functions when it's worth the complexity cost).
- In functions that have a hard dependency on the parent - accept objects over primitives: eg `update(user: User)` over `update(id: string)` - do not conceal the origin.

Extra guidelines:
- If type inference is broken - comment with `// @ts-expect-error #bad-infer {reason}` - explain when it's worth trying to remove this exception. Same for Python. Be extremely short.
- Avoid non-KISS constructs like `reduce` or list comprehensions. Unless they significantly improve readability.
- Use (or create) wrappers to declutter bad third-party APIs, eg Playwright.

#### FAANG-level defensive code in an MVP is tech debt

- Focus on readable code, instead of handling hypothetical edge-cases.
- Fail fast - in improbable cases use `assert` or `!`, not complex error handling.
- Assume backend enforces invariants. Use `?.` chaining, not defensive `if`s.
- Never hide errors - send to Sentry and users. Exceptions are only the _impactful_ security concerns.

#### TypeScript

- Use `??` for null/undefined. Never use `||` (preserves falsy).
- No inline conditionals: `if (x) action();` write as `if (x) {\n action()\n }`.
- Functions are declared as `function`, not `const`.
- Never use `export default`. Except when react-router requires it.
- No destructuring - it obscures the origin. The only exceptions:
    - Single-per-function `useApolloQuery`.
    - React soft-mandatory API: `useTransition`, `useState`, etc.

## Tags

Code blocks can be marked with:
- <TagLg>#AI</TagLg> - not 100% reviewed LLM code => deceiving and dangerous.
- <TagLg>#AI-slop</TagLg> - trash to drop.
- <TagLg>#bad-infer</TagLg> - broken TS type inference. Over time we'll check if it was fixed.
- <TagLg>#prob-redundant</TagLg> - but removal isn't a priority.
- <TagLg>#draft</TagLg> - not tested, not working, or not ready for prod.
- <TagLg>#E2E-fails</TagLg> - may be in Git commit body, because we still lack E2E CI.
