### Site selection: `VITE_SITE` is a Docker build ARG

`VITE_SITE` and `DJANGO_ENV` are baked into the `client` image at the JS build time.

See `client/Dockerfile` and `devops/docker.mise.toml` for the full ARG list (`VITE_GTM_ID`, `VITE_SERVER_URL`, Sentry, Algolia, etc).

To deploy a different `VITE_SITE` -> rebuild.

The `client` images are split on 4 names by `VITE_SITE` and `DJANGO_ENV`:
- NHA (`VITE_SITE = ""`):
    - `ghcr.io/.../client:$tag` - prod.
    - `ghcr.io/.../client-stage:$tag`.
- PG (`VITE_SITE = "pg"`):
    - `ghcr.io/.../client-pg:$tag` - prod.
    - `ghcr.io/.../client-pg-stage:$tag`.

### Algolia setup

I recommend creating `ALGOLIA_API_KEY` and `ALGOLIA_SEARCH_API_KEY` that are limited to `Indices: *_prod*` in <LinkExt href="https://dashboard.algolia.com/account/api-keys/restricted">the key settings</LinkExt> instead of using the global Write/Search API keys.

After the first deploy, connect to the `server` docker container, and run:

## Upgrades

Watch out for the `BREAKING CHANGE` notes in the release descriptions - it's specifically for self-hosting.

You may use `:latest` docker tag, but I advise to hardcode the non-breaking release eg `1.1`, as the two last numbers are for non-breaking changes, ie `1.1.25.3` won't break your deployment.


## Caveats

### Vite build hardcodes build-time envs

During the `docker build` step, the `pnpm exec react-router build` hardcodes `VITE_SERVER_URL` (and similar envs),
ie it's going to ignore the runtime envs as `SERVER_URL` and others.

### Unlike `client/` the `docs/` Docker image uses 6 runtime envs

3x2 Algolia envs: the app id and the search key (twice) - 3 for PG and 3 for NHA.


## Algolia

For envs as `stage` the API key needs to match the `*_stage*` indexes.
- Even if you need eg only `jobs_stage*` - it tries to modify `posts_stage` and others.
