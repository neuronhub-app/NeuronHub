## Why nested Popovers don't work (depth>0 → plain Code)

The spec says "Nested Popover" (Term inside Term's help).
Chakra docs show nested Popover works if the inner one
skips `<Portal>`. However two problems block it here:

1. Circular glossary: backend.help references `<Term id="frontend">`
   and frontend.help references `<Term id="backend">`. Rendering
   both as Popovers creates infinite recursion during SSR
   (react-dom-server stack overflow). Lazy `help()` fns solved
   the eager-eval, but SSR still recurses infinitely since each
   Popover body renders the other Term's full Popover tree.

2. MDX paragraphs: Terms appear inside `<p>`. Popover.Positioner
   renders `<div>`. During SSR, div-in-p is invalid HTML and
   causes hydration errors. `lazyMount unmountOnExit` fixes this
   for depth-0 (popover DOM only mounts on hover, after hydration).
   But a nested Popover inside the already-open parent would
   SSR-render the inner Positioner div immediately.

Decision: depth>0 Terms render as styled `<Code>` without
Popover. The dotted underline decoration still marks them as
glossary terms visually.

## Key decisions

- Glossary `help` values are `() => ReactNode` (lazy) to avoid
  circular JSX evaluation at module scope.
- `TermDepthCtx` React context tracks nesting. depth>0 = no Popover.
- `lazyMount unmountOnExit` prevents div-in-p SSR hydration error.
- `env.ts` DOCS_PORT/DOCS_PORT_E2E need defaults because
  `createEnv` evaluates in browser where only VITE_* vars exist.
  Mise is source of truth; defaults are just browser-eval safety.
- PW hover tests wait for `data-scope=popover` attr (= hydrated)
  before hover, fixing flakiness under 16 workers.
- Console error test filters pre-existing color-mode
  hydration-mismatch (className="light" on `<html>`).
